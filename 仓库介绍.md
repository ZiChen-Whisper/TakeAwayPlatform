> 本文档的目标读者是**完全不熟悉本项目的初学者**，将从零开始介绍项目的每一个目录、每一个文件的作用和内部逻辑。

---

## 一、项目概述

**TakeAwayPlatform** 是一个用 **C++** 编写的外卖平台后端服务程序。它的主要功能是：

1. 启动一个 HTTP 服务器，监听在 9090 端口
2. 接收客户端发来的 HTTP 请求（比如查看菜单、下单）
3. 连接 MySQL 数据库，对数据进行增删查改
4. 将处理结果以 JSON 格式返回给客户端

一句话总结：**它是一个外卖系统的"后台大脑"**，负责接收请求、处理业务、操作数据库。

## 五、每个文件的详细介绍

### 5.1 `CMakeLists.txt` — 构建配置

这是 CMake 的配置文件，**告诉编译器怎么把 C++ 源码编译成可执行程序**。

```cmake
# 关键内容解读：

# 第6行：设置用 C++17 标准编译
set(CMAKE_CXX_STANDARD 17)

# 第45行：把 jsoncpp 的源码编译成一个静态库 jsoncpp_lib
add_library(jsoncpp_lib STATIC ${JSONCPP_SOURCES})

# 第60行：编译出最终的可执行文件，名字叫 TakeAwayPlatform
add_executable(${PROJECT_NAME} ${SOURCES})

# 第63-71行：链接所有需要的库
target_link_libraries(${PROJECT_NAME}
    PRIVATE 
        jsoncpp_lib      # JSON解析库
        mysqlcppconn     # MySQL连接库
        ssl crypto z     # 加密和安全库
        pthread          # 多线程库
)

# 第50行：MySQL动态库的实际加载路径
link_directories(/opt/TakeAwayPlatform/bin/lib)

# 第78行：把 config/ 安装到部署目录
install(DIRECTORY ${CONFIG_DIR}/ DESTINATION ${CMAKE_INSTALL_PREFIX}/config)
```

**大白话解释**：这个文件就像一份"施工图纸"，告诉编译器：用 C++17、去哪里找头文件、编译哪些源文件、链接哪些库，最后生成一个叫 `TakeAwayPlatform` 的程序。

### 5.4 `include/log.h` — 日志系统

这是一个 **header-only** 的日志类（全部实现都在头文件里，没有单独的 .cpp 文件）。

核心设计：
- **单例模式** (`getInstance()`)：整个程序只有一个日志对象，避免重复创建
- **线程安全**：用 `std::mutex` 互斥锁保护，多个线程同时写日志不会混乱
- **四级日志**：DEBUG（调试） < INFO（信息） < WARNING（警告） < ERROR（错误）
- **同时输出到控制台和文件**：方便开发时看屏幕，也方便部署后查日志文件
- **默认日志路径**：`/opt/TakeAwayPlatform/logs/takeaway.log`

**大白话解释**：就像程序的一本"日记本"，记录程序运行时发生的事情。比如"服务器启动了"、"数据库连接成功"、或者"出错了"。

---

### 5.5 `include/task_queue.h` — 任务队列

也是 **header-only** 设计。实现了一个线程安全的任务队列：

```cpp
class TaskQueue {
    void push(Task task);  // 往队列里放一个任务
    Task pop();            // 从队列里取一个任务（如果队列为空就等着）
    bool empty() const;    // 检查队列是不是空的
};
```

核心机制：
- 用 `std::queue` 存储任务（任务就是一个函数 `std::function<void()>`）
- 用 `std::mutex` 保证多线程安全（同时只有一个线程能操作队列）
- 用 `std::condition_variable` 实现"没任务就睡觉，有任务就叫醒"

**大白话解释**：想象一个排队系统。"生产者"往队列里放任务，"消费者"从队列里取任务执行。队列空了消费者就休息，有新任务来了就被叫醒继续干活。

---

### 5.6 `include/thread_pool.h` — 线程池

也是 **header-only** 设计。在任务队列的基础上实现线程池：

```cpp
class ThreadPool {
    ThreadPool(size_t thread_count);  // 创建时指定多少个工作线程
    void enqueue(F&& task);           // 往线程池投递一个任务
};
```

核心机制：
- 构造时创建 N 个工作线程（默认等于 CPU 核心数）
- 每个工作线程不断从 `TaskQueue` 里取任务执行
- 析构时通知所有线程停止，等它们结束

**大白话解释**：线程池就是预先"招聘"好的一群工人（线程）。来一个任务就交给一个空闲的工人去处理，不用每次临时招人（创建线程很费时间）。这是提高服务器性能的关键设计。

### 5.7 `src/main.cpp` — 程序入口

**这是整个程序的起点**，阅读代码应该从这里开始。按执行顺序：

2. **初始化日志**（第44-47行）：设置日志级别为 DEBUG，输出到默认文件
3. **创建 RestServer**（第49行）：传入配置文件路径，初始化数据库连接池

**大白话解释**：`main.cpp` 就像公司的大门——程序从这里开始运行。它的工作流程是：准备环境 → 启动服务 → 等待关闭信号 → 安全退出。

### 5.8 `src/http/rest_server.h` + `rest_server.cpp` — HTTP 服务器

这是项目**最核心的类**，负责处理所有 HTTP 请求。

#### 类的成员变量

| 变量 | 作用 |
|------|------|
| `server` | cpp-httplib 的 HTTP 服务器对象 |
| `threadPool` | 线程池，处理并发请求 |
| `dbPool` | 数据库连接池（预创建一堆数据库连接，随用随取） |
| `dbConfig` | 数据库配置信息 |
| `isRunning` / `stopRequested` | 原子布尔标志，控制服务器启停 |
| `serverThread` | 运行服务器的独立线程 |

#### 核心方法

**`start(int port)`**（第33行）：启动服务器
- 创建一个新线程，在后台执行 `run_server()`
- 新线程里调用 `server.listen("0.0.0.0", port)` 开始监听

**`setup_routes()`**（第183行）：设置 URL 路由（最关键的方法！）

定义了三个 API 接口：

| 方法 | 路径 | 作用 |
|------|------|------|
| GET | `/` | 返回"TakeAwayPlatform is running!"，确认服务活着 |
| GET | `/health` | 健康检查，返回"OK"或"SHUTTING_DOWN" |
| GET | `/menu` | 查询所有菜品，投递给线程池处理 |
| POST | `/order` | 创建订单，解析请求中的 JSON 数据并处理 |

**大白话解释**：`rest_server` 就像餐厅的前台服务员。客人（客户端）进来后：说"你好"（GET /）→ 回复"欢迎光临"；说"看看菜单"（GET /menu）→ 去数据库查菜品返回；说"我要下单"（POST /order）→ 解析订单内容处理。

---

### 5.9 `src/database/db_handler.h` + `db_handler.cpp` — 数据库处理器

负责与 MySQL 数据库的所有交互。

#### 核心方法

**`connect(config)`**（第55行）：连接数据库
- 通过 `tcp://host:port` 格式连接 MySQL
- 连接成功后切换到指定数据库（`setSchema`）

**`query(sql)`**（第16行）：执行 SQL 查询
- 创建 Statement 对象 → 执行 SQL → 获取结果集 → 转换成 JSON 返回

**`parse_result(result)`**（第82行）：把 MySQL 结果转成 JSON
- 遍历结果集的每一行、每一列
- 根据列的数据类型（字符串/整数/浮点数/布尔）分别处理
- 最终返回 `Json::Value` 数组

**`is_connected()`**（第34行）：检查连接是否存活（执行 `SELECT 1` 测试）

**`reconnect()`**（第49行）：断开并重新连接

**大白话解释**：`db_handler` 相当于一个数据库"遥控器"。你给它一句 SQL 语句（比如 `SELECT * FROM menu_items`），它帮你发给 MySQL，然后把你查到的结果打包成 JSON 格式还给你。

## 六、部署目录详细说明 (`/opt/TakeAwayPlatform/`)

```
│           ├── authentication_kerberos_client.so
│           └── ...
└── config/
    └── config.json               ← 运行时配置文件
```

### 重要概念解释

**动态库（.so 文件）是什么？**

程序运行时需要的"零件"，不是编译时嵌进去的，而是启动时去磁盘上找。好处是多个程序可以共享同一个库文件，节省空间。`libmysqlcppconn8.so` 就是 MySQL 连接功能的"零件库"。

**为什么库文件放在 `/opt/TakeAwayPlatform/bin/lib/`？**

因为 CMakeLists.txt 第50行硬编码了这个路径：
```cmake
link_directories(/opt/TakeAwayPlatform/bin/lib)
```
程序启动时会去这个路径找动态库。这就是仓库和部署目录分离的设计——开发在 `~/TakeAwayPlatform/`，运行在 `/opt/TakeAwayPlatform/`。

## 七、程序的启动流程（完整串联）

用一张流程图总结整个程序的运行逻辑：

```
1. 用户运行 ./TakeAwayPlatform
         ↓
2. main() 开始执行
         ↓
3. 注册信号处理函数（Ctrl+C 能安全退出）
         ↓
4. 初始化日志系统（日志写到 /opt/TakeAwayPlatform/logs/takeaway.log）
         ↓
5. 创建 RestServer 对象
   ├── 读取 /opt/TakeAwayPlatform/config/config.json
   ├── 获取数据库配置
   └── 创建数据库连接池（预建10个连接）
         ↓
6. 启动 HTTP 服务器（在后台线程监听 9090 端口）
         ↓
7. 主线程进入等待状态（等 Ctrl+C）
         ↓
8. HTTP 请求来了 → cpp-httplib 接收 →
   ├── GET /          → 直接返回 "TakeAwayPlatform is running!"
   ├── GET /health    → 返回健康状态
   ├── GET /menu      → 投递到线程池 → 从连接池取DB连接 → 执行SQL → 返回JSON
   └── POST /order    → 投递到线程池 → 解析JSON → 处理订单 → 返回结果
         ↓
9. 用户按 Ctrl+C → 信号处理函数设置 running=false
         ↓
10. 主线程醒来 → 调用 restSrv.stop() → 等待服务器线程结束
         ↓
11. 程序安全退出
```

## 八、如何编译和运行

### 编译

```bash
cd ~/TakeAwayPlatform
mkdir -p build && cd build
cmake ..
make
```

编译成功后，可执行文件在 `build/TakeAwayPlatform`。

### 部署到运行目录

```bash
# 把编译好的程序和配置复制到部署目录
cp build/TakeAwayPlatform /opt/TakeAwayPlatform/bin/
cp config/config.json /opt/TakeAwayPlatform/config/
```

### 运行

```bash
cd /opt/TakeAwayPlatform/bin
./TakeAwayPlatform
```

按 `Ctrl+C` 安全退出。

## 九、关键概念速查表

| 术语 | 通俗解释 |

| **HTTP 服务器** | 接收网页请求并返回数据的程序 |
